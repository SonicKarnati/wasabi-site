# Accessibility and Trust Checklist

## Accessibility

### Semantic Page Structure

- One H1 per page.
- Logical heading order.
- Landmarks for header, nav, main, footer, and forms.
- Lists and tables use semantic markup when implemented.

### Alt Text Strategy

- Real photos need descriptive alt text.
- Decorative lines or textures should have empty alt text.
- Partner logos need partner name and relationship context if approved.
- Student/community photos require rights and consent before use.

### Color Contrast

- Body text on cream/white must meet WCAG contrast.
- Text on forest dark should be white or cream.
- Avoid green body text on forest dark.
- Do not rely on color alone for proof status.

### Keyboard Navigation

- All links, buttons, cards, accordions, nav menus, and forms must be keyboard accessible.
- Focus order should follow visual order.
- Mobile nav should be operable with keyboard and Escape.

### Reduced Motion

- Respect reduced-motion preferences.
- No animation-dependent information.
- Avoid motion that distracts from reading.

### Focus States

- Focus states must be visible on all interactive elements.
- Focus states should have strong contrast and not rely on subtle color alone.

### Forms

- Every field needs a visible label.
- Required fields should be marked clearly.
- Error messages should identify the field and the issue.
- Success/failure states should be clear.
- Consent/privacy text is required before implementation.

### Mobile Usability

- Touch targets should be at least 44px where possible.
- Cards and proof strips should stack cleanly.
- Text should not overflow buttons or cards.
- Navigation should stay simple.

### Performance

- Optimize images before production.
- Avoid heavy animation libraries unless justified.
- Do not use large PDFs as page assets.
- Use simple graphic treatments before heavy visual effects.

## Trust

### Claim Safety

- No unsupported claims.
- No invented impact numbers.
- No fake partner logos.
- No fake testimonials.
- No exaggerated rankings.
- No third-party quotes without permission and context.
- No public-sector naming without approval.

### Proof Handling

- Separate verified impact, projected/target metrics, program design, and placeholders.
- Every metric needs a source and approval status.
- Case studies require consent and source documentation.
- Program status must be clear: active, completed, proposed, future, or unclear.

### Contact Transparency

- Use one official public domain and email once confirmed.
- Do not publish mixed contact details without routing logic.
- Contact form should make inquiry type clear.
- Do not imply enrollment, funding, or partnership confirmation from a form submission.

### Organization Description

- Explain Wasabi plainly.
- Keep human capability central.
- Avoid generic "AI revolution" language unless made specific.
- Make Bharat-native context clear without unsupported national-scale claims.

### Partner and Advisor Trust

- Only show partner logos with permission.
- Only publish advisor/team names with current titles and approval.
- Use text-only placeholders until permissions are confirmed.

### Human Language

- Write for people, not personas.
- Avoid cold corporate language.
- Avoid fake motivational language.
- Avoid overpromising academic, financial, or career outcomes.

### Visual Trust

- Use restraint, whitespace, readable typography, and calm color.
- Avoid sci-fi AI visuals.
- Avoid crowded brochure dumps.
- Avoid generic stock people.
- Avoid excessive motion and decorative effects.

## Pre-Launch Trust Checklist

- [ ] Official domain/contact confirmed.
- [ ] Every metric has source and approval.
- [ ] Partner logos approved.
- [ ] Third-party quotes approved.
- [ ] Student/community stories consented.
- [ ] Team/advisor bios approved.
- [ ] Program statuses confirmed.
- [ ] Privacy and consent language reviewed.
- [ ] Accessibility review completed.
- [ ] No projected metrics shown as achieved impact.
